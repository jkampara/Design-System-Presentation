02 — The Landscape

Design Systems at Salesforce

Multiple design systems exist across Salesforce, each built on shared foundations but diverging at the component layer. Here’s where we sit — and what’s missing.

Parent System
🏛
SLDS 2 — Cosmos
Salesforce Lightning Design System. Provides foundational design tokens and a component library. Some subsystems consume both foundations and components; others inherit only the foundations they need.
Colors Typography Spacing Iconography Components

Subsystem
🌐
WES — Website Experience Subsystem
Designed for a unified experience across 108+ Salesforce marketing sites and 417,000+ pages. Built to unify post-acquisition brands (Mulesoft, Tableau, Quip) under one cohesive look. Inherits select SLDS foundations (colors, typography) but builds its own component library on LWC.
Foundations Component Library (LWC) Figma Design System Docs

Consumer
🏗
DMT PageBuilder
Uses WES color, typography, and guidelines to stay visually consistent. Cannot use WES components — PageBuilder runs on Nunjucks with a different rendering pipeline. Builds its own component set.
Nunjucks Own Components
⚠ Tech stack mismatch — WES LWC components are not compatible
Consumer
⚙
MFEs (IDX, AFE, OGX)
Follow WES guidelines for visual consistency but use React. Maintain a separate library “WES Components” — managed by MFE engineers, not WES developers.
React Own Components
⚠ Own component library — diverges over time without WES governance
🎨 Both PageBuilder and MFEs share a single Figma Design System file for visual consistency
