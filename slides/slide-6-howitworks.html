🗂 Parse Intent → Schema
The LLM reads the schema to discover which page types, section patterns, and component combinations are available. Instead of inventing a layout, it maps “product launch with hero + grid + CTA” to a known template entry — the intent is resolved to a concrete recipe name before anything is built.
🧩 Select Template → Recipes
Recipes define canonical page skeletons with typed slots. The LLM selects the matching recipe and reads its blade definitions — each slot declares what it expects (image ratio, text length, CTA style) so the page structure is fixed and content is variable.
📐 Assemble → Component Guidelines
No pre-built recipe exists, so the LLM falls back to component-level guidelines. Each component documents its allowed sizes, spacing, combination rules, and content requirements. The LLM assembles a new layout from these primitives — acting as a proposer, not just a populator.
🎨 Compose Page → Tokens
Tokens provide every visual decision as structured values. The LLM references semantic tokens like --color-fill-accent instead of guessing hex codes. Theme switching (Day/Night) is a single attribute change, not a rewrite.
⛔ Validate Output → Constraints
Constraints enforce hard rules — no invented hex codes, hero-only Display 1, pill-shaped buttons. Coverage data tells the validator what is tokenized vs. what needs human judgment. Uncovered concerns trigger a hard stop instead of hallucination.
🗂 Schema
The schema is the design system’s table of contents — a machine-readable registry of every page type, section pattern, and component available. When the LLM receives a user prompt, it checks the schema first to resolve intent into a known template name.
Example — schema.json
{
  "pageTypes": {
    "product-launch": {
      "description": "Hero + features grid + CTA banner",
      "allowedBlades": ["hero", "feature-grid", "cta-banner"],
      "intents": ["launch", "announce", "promote product"]
    },
    "event-landing": {
      "description": "Hero + agenda + speakers + register",
      "allowedBlades": ["hero", "schedule", "speaker-grid", "cta-banner"],
      "intents": ["event", "conference", "webinar"]
    },
    "content-hub": {
      "description": "Category nav + card grid + pagination",
      "allowedBlades": ["filter-bar", "card-grid", "pagination"],
      "intents": ["blog", "resources", "library"]
    }
  }
}
🧩 Recipes
Recipes are canonical page and component skeletons with typed slots. They define structure, not content — any consumer fills the slots with real data. The LLM picks a recipe, then populates each blade with CMS content and DAM assets.
Example — Product Launch recipe
{
  "recipe": "product-launch",
  "blades": [
    {
      "type": "hero",
      "slots": {
        "headline": { "maxLength": 60 },
        "subhead":  { "maxLength": 120 },
        "image":    { "ratio": "16:9" },
        "cta":      { "style": "primary" }
      }
    },
    {
      "type": "feature-grid",
      "slots": {
        "columns": 3,
        "card": { "icon": true, "title": true, "body": true }
      }
    },
    {
      "type": "cta-banner",
      "slots": { "headline": true, "cta": { "style": "primary" } }
    }
  ]
}
📐 Component Guidelines
When no recipe matches, the LLM falls back to component-level guidelines. Each component documents its sizing, spacing, allowed nesting, and combination rules. The LLM assembles a new page from these primitives, constrained by what each component permits.
Example — component guidelines
{
  "components": {
    "hero": {
      "maxPerPage": 1,
      "position": "first-blade-only",
      "slots": {
        "headline": { "maxLength": 60, "font": "display-3" },
        "image": { "ratio": ["16:9", "21:9"] }
      }
    },
    "card": {
      "allowedIn": ["grid", "carousel", "split"],
      "variants": ["default", "icon-top", "image-top"],
      "spacing": { "gap": "space-content-md" }
    },
    "cta-banner": {
      "position": "last-blade-preferred",
      "maxButtons": 2,
      "buttonStyles": ["primary", "secondary"]
    }
  },
  "nesting": {
    "grid": { "minColumns": 2, "maxColumns": 4 },
    "split": { "ratio": ["50:50", "60:40", "40:60"] }
  }
}
🎨 Tokens
Tokens encode every visual decision — color, typography, spacing, radii — as structured key-value pairs published as CSS custom properties and JSON. Semantic tokens reference primitives, so switching themes requires zero code changes.
Example — Semantic color tokens
/* Day theme */
[data-theme="day"] {
  --color-fill-accent:      #066AFE;
  --color-on-fill-accent:   #FFFFFF;
  --color-surface-1:        #FFFFFF;
  --color-on-surface-1:     #181818;
  --color-border-accent:    #022AC0;
}

/* Night theme — same names, different values */
[data-theme="night"] {
  --color-fill-accent:      #90D0FE;
  --color-on-fill-accent:   #001D3D;
  --color-surface-1:        #1A1A2E;
  --color-on-surface-1:     #F0F0F0;
  --color-border-accent:    #5B9CF6;
}
⛔ Constraints
Constraints combine hard rules with coverage data. Rules define what must always be true. Coverage declares what’s tokenized (safe to automate) vs. what needs human judgment. The validator reads both as a single checklist before any output ships.
Example — constraints + coverage
{
  "rules": [
    { "id": "no-invented-colors",
      "severity": "error",
      "check": "All color values must resolve to a token" },
    { "id": "hero-display-only",
      "severity": "error",
      "check": "Display 1 typeface restricted to hero blades" },
    { "id": "button-shape",
      "severity": "warning",
      "check": "Buttons use radius-pill unless overridden" }
  ],
  "coverage": {
    "tokenized": ["color", "typography", "spacing", "radii"],
    "uncovered": {
      "illustration-style": "ask designer",
      "motion-curves": "not yet tokenized",
      "data-viz-palette": "pending review"
    }
  }
}
📝 Content Agent
The Content Agent is responsible for populating every text slot in the page. It pulls existing copy, product descriptions, and messaging from the CMS, then generates new copy where gaps exist — adhering to brand voice rules, tone guidelines, and character limits defined by the recipe’s slot constraints.
Capabilities
{
  "sources": ["CMS", "product catalog", "brand voice DB"],
  "actions": [
    "Fetch existing copy by page context",
    "Generate headlines within slot maxLength",
    "Adapt tone per audience segment",
    "Localize content for region variants"
  ],
  "constraints": {
    "maxHeadline": 60,
    "maxSubhead": 120,
    "voiceProfile": "salesforce-professional"
  }
}
🖼 Media Agent
The Media Agent retrieves images, illustrations, and icons from the Digital Asset Manager (DAM). It matches assets to layout slots based on required aspect ratios, resolution, and content tags — ensuring every visual is on-brand and correctly sized before it reaches the page.
Capabilities
{
  "sources": ["DAM", "brand asset library", "icon set"],
  "actions": [
    "Search assets by content tags and context",
    "Match aspect ratio to slot requirements",
    "Auto-crop and resize for target breakpoint",
    "Fallback to placeholder if no asset found"
  ],
  "constraints": {
    "maxHeight": "620px",
    "formats": ["webp", "svg", "png"],
    "requiredAlt": true
  }
}
🔍 SEO Agent
The SEO Agent optimizes the final page for search engine visibility. It validates heading hierarchy, generates meta descriptions, ensures proper semantic HTML structure, and checks against keyword targets — all before the page ships.
Capabilities
{
  "actions": [
    "Validate single H1 per page",
    "Generate meta title and description",
    "Check heading hierarchy (H1 → H2 → H3)",
    "Add structured data / JSON-LD",
    "Optimize image alt text for keywords",
    "Verify canonical URL and Open Graph tags"
  ],
  "constraints": {
    "metaTitleMax": 60,
    "metaDescMax": 155,
    "singleH1": true
  }
}
