04 — Current State

Where Are We Today

Before we build the future, we need to be honest about what exists. Our design system landscape is fragmented across systems, tech stacks, and CSS practices — making agent adoption harder than it needs to be.

📦 Scattered UI Systems

Experience Library covers ~80% of components through blades and shared UI, but major MFEs operate on entirely separate systems.

Experience Library IDX AFE OGX

Each MFE maintains its own component set, duplicating effort and diverging from the core system.

⚙ No Singular Tech Stack

Different teams use different frameworks and rendering approaches. There’s no single stack that all surfaces build on.

Intent Converge on a unified tech stack across all surfaces

Migration and new feature work is slow because every surface requires different implementation patterns.

🎨 CSS Without Semantic Layers

We don’t use semantic variables or global aliases today. Raw hex values and one-off styles are scattered across the codebase.

Today color: #032D60; background: #FFFFFF;
Needed color: var(--color-on-surface-1); background: var(--color-surface-1);

No theming possible. Multi-brand adoption (Salesforce+, Slack) requires rewriting CSS, not flipping an attribute.

📋 Too Many Blades & Cards

A long, growing list of blade and card variants makes maintenance costly and introduces ambiguity for both humans and AI.

25+ Blade variants
15+ Card types

More choices = more error-prone. LLMs struggle to pick the right component when the catalog is ambiguous and overlapping.

📖 Non-Interactive Storybook

Our Storybook today is a static visual reference — useful for humans browsing, but not built for programmatic consumption or automated workflows.

  • No machine-readable test harnesses that LLMs can invoke to validate output
  • No interaction stories that demonstrate state transitions, props combinations, or edge cases
  • No API for agents to query component capabilities, required props, or slot structures
  • Visual regression tests exist but aren’t wired into any agent feedback loop

An interactive, API-driven Storybook would let LLMs test renders, verify constraints, and self-correct before shipping code.

💸 No Vendor Documentation Strategy

External vendors build custom open blades from scratch because there’s no documentation guiding them to reuse our existing component library.

Cost Every vendor-built blade is a one-off — expensive to create, review, and maintain long-term

With clear component docs and composable APIs, vendors could assemble pages from our system — faster delivery, lower cost, brand-safe output.
